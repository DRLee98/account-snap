## 범위

**MVP에는 미포함.** 단, MVP 데이터 모델([[DAT-001]])이 동기화 가능한 구조로 설계되어 있어야 추후 추가 시 마이그레이션 부담이 없음. 이 문서는 그 가이드라인을 명시.

## 동기화 전제

- 단일 사용자, 다중 디바이스 시나리오 (예: 폰 + 태블릿)
- 충돌은 드물다고 가정 → last-write-wins (updatedAt 기준)
- 오프라인 우선: 로컬이 SoT(source of truth), 서버는 백업/공유 채널

## 후보 백엔드

| 옵션 | 장점 | 단점 |
|------|------|------|
| **Supabase** | Postgres + Auth + Realtime 즉시 사용, 무료 티어 넉넉 | 자체 인프라 운영 필요 |
| **Firebase Firestore** | 오프라인 동기화 내장, 모바일 SDK 성숙 | 쿼리 제약, 비용 예측 어려움 |
| **iCloud + Google Drive** | 사용자 비용 0, 계정 시스템 불필요 | 양 플랫폼 SDK 다름, 개발 부담 |

→ MVP 이후 사용자 피드백 보고 선정. 1차 추천은 **Supabase** (무료 + Postgres + 추후 확장 용이).

## 동기화 프로토콜 (예시)

```
1. 마지막 sync 시각 이후 변경된 로컬 레코드 → 서버 push
2. 마지막 sync 시각 이후 서버에서 변경된 레코드 → 로컬 pull
3. 같은 id가 양쪽에서 변경된 경우 → updatedAt이 큰 쪽 채택
4. deletedAt이 set된 레코드는 양쪽에서 동일하게 soft delete 유지
```
